NOTE on NEXT_PUBLIC_* variables:
  These are build-time only in Next.js. They are baked into the image during
  `docker build`. The ConfigMap documents expected values but does NOT inject
  them at runtime. To change NEXT_PUBLIC_* values, rebuild the image.
  The frontend-config ConfigMap carries runtime env vars only (PORT, HOSTNAME, NODE_ENV).

<task type="auto">
  <name>Task 1: Create frontend Deployment and Service manifests</name>
  <files>k8s/base/frontend/deployment.yaml, k8s/base/frontend/service.yaml</files>
  <action>
Create directory `k8s/base/frontend/`.

Create `k8s/base/frontend/deployment.yaml`:
- apiVersion: apps/v1, kind: Deployment
- metadata: name `frontend`, namespace `multica`, labels `app.kubernetes.io/name: frontend` and `app.kubernetes.io/part-of: multica`
- spec.replicas: 2
- spec.selector.matchLabels: `app.kubernetes.io/name: frontend`
- spec.strategy: type RollingUpdate, maxSurge 1, maxUnavailable 0
- Pod template labels: same as selector plus `app.kubernetes.io/part-of: multica`
- spec.template.spec.serviceAccountName: frontend-sa
- spec.template.spec.terminationGracePeriodSeconds: 30
- Single container named `frontend`:
  - image: `frontend:latest` (placeholder — overlays replace with real registry)
  - ports: containerPort 3000, name `http`
  - envFrom:
    - configMapRef: name `frontend-config`
  - env (individual vars not in ConfigMap):
    - name: REMOTE_API_URL, value: "http://backend:8080" (K8s service DNS for API proxy)
  - resources:
    - requests: cpu 100m, memory 256Mi
    - limits: cpu 500m, memory 512Mi
  - livenessProbe:
    - httpGet: path /, port 3000
    - periodSeconds: 10
    - failureThreshold: 3
  - readinessProbe:
    - httpGet: path /, port 3000
    - periodSeconds: 5
    - failureThreshold: 3
  - startupProbe:
    - httpGet: path /, port 3000
    - periodSeconds: 2
    - failureThreshold: 30

Do NOT add preStop hook to frontend — Node.js doesn't need connection draining like the Go backend.

Create `k8s/base/frontend/service.yaml`:
- apiVersion: v1, kind: Service
- metadata: name `frontend`, namespace `multica`, labels `app.kubernetes.io/name: frontend` and `app.kubernetes.io/part-of: multica`
- spec.type: ClusterIP
- spec.selector: `app.kubernetes.io/name: frontend`
- spec.ports: port 3000, targetPort 3000, protocol TCP, name `http`

Follow the EXACT same YAML formatting and structure as the Phase 1 postgres Service and Phase 2 backend Service.
  </action>
  <verify>
    <automated>kustomize build k8s/base/frontend/ 2>&1 | head -5</automated>
  </verify>
  <done>
    - deployment.yaml exists with all specified fields
    - service.yaml exists matching established service pattern
    - Both files are valid YAML that kustomize can process
  </done>
